# swg-panel

Self-hosted панель управления небольшим сервисом WireGuard / AmneziaWG на одном или нескольких входных серверах. Панель — единственный источник истины о пирах; каждый входной сервер (**узел**) синхронизируется с ней по **исходящему HTTPS** и приводит свой набор пиров к назначенному — поэтому узлам не нужен входящий доступ, SSH или rsync.

Рассчитано на небольшие доверенные развёртывания — десятки пользователей, несколько серверов — с прицелом на многозвенную схему (входной узел на краю → удалённый выход).

## Как это работает

- **Панель** (управляющий слой) — отдаёт веб-интерфейс, собственный TLS + логин и API синхронизации узлов. Хранит реестр (ваши пиры) и список узлов. Чистый Python, без базы данных.
- **Узел** (входной сервер) — запускает `swg-noded`, который каждые несколько секунд отправляет снимок своего интерфейса панели, получает набор пиров, которые на нём *должны* быть, и приводит интерфейс к этому состоянию через `swg-agent` (добавляя/удаляя пиры на «живом» wg/awg-интерфейсе). Только исходящий HTTPS.
- **Декларативно** — вы меняете пиры в панели; узлы подтягиваются сами. Узел, пропустивший такт, восстановится при следующей синхронизации, а кратковременный сбой панели никогда не сотрёт пиры узла (узел применяет изменения только при корректном ответе).
- **Ключи остаются у владельца** — пары ключей пиров генерируются в вашем браузере. Приватный ключ попадает в конфиг/QR, который показывается **один раз**, и никогда не передаётся на сервер; панель хранит лишь публичный ключ, выданный IP и общий ключ (PSK), чтобы держать все узлы согласованными.

## Method × Role

Два независимых выбора.

**Method** — как устанавливается:
- **bare-metal** — службы systemd, через `install-host.sh` / `install-node.sh`.
- **docker** — `docker compose` с образами `swg-panel` / `swg-node`.

**Role** — что делает машина:
- **host** — только панель.
- **master** — панель *и* эта машина также входной сервер (подключается автоматически).
- **node** — только входной сервер.

Можно смешивать: Docker-панель с bare-metal-узлами, bare-metal `master` плюс дополнительные Docker-узлы и т. д.

## Компоненты

| компонент | где | назначение |
|---|---|---|
| `swg-panel-server` | хост | интерфейс + API синхронизации + реестр/список узлов; отдаёт собственный TLS + логин и панель статуса |
| `swg-noded` | каждый узел | отправляет снимки, получает нужный набор пиров, согласует локально — только исходящий HTTPS |
| `swg-agent` | каждый узел | применяет одно добавление/удаление пира на «живом» интерфейсе и в его `.conf` |
| `users.json` | хост | **реестр** — ваши пиры (публичный ключ, IP, PSK и на каких узлах живёт каждый) |
| `nodes.json` | хост | **список узлов** — имена, адреса и хеши токенов подключения |

Вручную `users.json` и `nodes.json` не редактируются — это делает интерфейс.

## Быстрый старт

Четыре однострочника — каждый **спрашивает всё необходимое**. Выберите способ (bare-metal или Docker) на каждой машине; можно смешивать.

### A — bare-metal (systemd)

**Панель** — спросит роль: master (панель + этот узел) или host (только панель)

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s host
```

**Узел** — спросит URL панели + ключ из Nodes → Add node

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s node
```

### B — Docker

**Панель** — установит Docker при необходимости, затем спросит роль (master: панель + этот узел, регистрируется автоматически · host: только панель) и домен + выбор TLS (логин генерируется автоматически, поменяете позже в панели)

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s docker host
```

**Узел** — спросит URL панели + ключ из Nodes → Add node

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s docker node
```

Откройте URL панели, войдите и добавляйте входные серверы через **Nodes → Add node** — он печатает точную команду и для bare-metal, и для Docker (ключ уже подставлен). Подробнее: [Установка панели](#установка-панели) · [Добавление узла](#добавление-узла) · [Docker](#docker).

**Обновление** любой машины на месте (определяет установленное автоматически):

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s update
```

## Установка панели

`install-host.sh` (запускается через `bootstrap.sh host` или напрямую из клона) ставит панель и, для `master`, локальный входной сервер. Две последовательные части — **Настройка панели**, затем **Настройка узла** (только master):

| часть | вопрос | смысл |
|---|---|---|
| Панель · 1 | **Role** | `master` (по умолчанию — панель + эта машина как входной сервер) или `host` (только панель) |
| Панель · 2 | **URL панели** | IP, хост или хост с подпутём (`vpn.example.com/swg`), чтобы разместить панель под существующим сайтом. По умолчанию — публичный IP этой машины. |
| Панель · 3 | **TLS** | `letsencrypt` (по умолчанию) · `cloudflare` · `cf15` · `selfsigned` · `skip` |
| Панель · 4 | **Режим отдачи** | `internal` (по умолчанию, самодостаточный) · `nginx` · `caddy` · `skip` |
| Панель | **Порт / админ** | порт (по умолчанию **443** для `internal`; unit добавит право на привязку для портов < 1024) и веб-логин — предлагает `admin` + 3 цифры; меняется позже в разделе **Account** |
| Узел · 1 | **Имя узла** | *(master)* имя этого узла (по умолчанию — hostname) |
| Узел · 2 | **Endpoint IP** | *(master)* публичный IP, который набирают клиенты (по умолчанию — определяется) |
| Узел · 3 | **Интерфейсы** | *(master)* управляет **всеми** найденными интерфейсами wg/awg (сканируя весь `/etc/amnezia` и `/etc/wireguard`); предложит **создать**, если их нет; нажмите **Enter**, чтобы продолжить, или **`new`**, чтобы добавить ещё. Новые интерфейсы получают серверный ключ, подсеть и автоматический forward + masquerade (`PostUp`/`PostDown`). |
| Узел · 4 | **Turn-proxy** | *(master)* опционально [vk-turn-proxy](https://github.com/cacggghp/vk-turn-proxy) — туннелирует wg/awg через TURN-серверы VK/Яндекс. Находит установленный прокси (по unit с `-listen`/`-connect`) и показывает его; **Enter** — пропустить, либо выбрать форк по **номеру** для установки (`WINGS-N`/`samosvalishe`/`kiper292`/`Moroka8` для Android, `anton48` для iOS). Скачивает релизный бинарник, **сам подставляет** порт `-connect` из настроенного интерфейса wg/awg (показывает все), **генерирует 64-hex wrap-ключ** с флагами форка (`-wrap-srtp`/`-wrap`/`-wrap-mode`; у `kiper292` их нет) и записывает `listen`/`connect`/`wrap_key` для панели и клиентских конфигов. На **bare-metal** каждый прокси — systemd-сервис, цель которого хранится в `EnvironmentFile`, поэтому правка из панели лишь переписывает его и перезапускает (без `daemon-reload`); в **Docker** это соседний **контейнер** (`swg-turn-*`, управляется через смонтированный сокет Docker — без `--privileged`/`--pid=host`). |

**TLS:**
- **letsencrypt** (по умолчанию) — настоящий сертификат через `acme.sh` (HTTP-01 standalone для `internal`/`caddy`, webroot за `nginx`); нужен доступный порт 80.
- **cloudflare** — настоящий сертификат через DNS-01; **порт 80 не используется**. Токену нужны права `Zone:DNS:Edit` + `Zone:Read`.
- **cf15** — **Origin**-сертификат Cloudflare, **15 лет**, через CF API (нужен **API-токен** с `Zone` → `SSL and Certificates` → `Edit`; устаревший Origin CA Key больше не используется). ⚠️ Доверенный только **за прокси Cloudflare** (оранжевое облако) — при прямом обращении к origin браузер покажет недоверенный сертификат. Продление не нужно 15 лет.
- **selfsigned** — мгновенно; браузер один раз предупредит. Подходит для старта или за туннелем.
- **skip** — свой сертификат (`CERT_FULLCHAIN`/`CERT_KEY`) или терминирование TLS снаружи. Без сертификата панель/прокси отдаёт обычный HTTP.

**Режимы отдачи:** `internal` самодостаточный (свой TLS + логин); `nginx` / `caddy` — обратный прокси, панель на loopback (TLS терминирует прокси, подпуть становится блоком `location`/`handle`); `skip` оставляет панель на loopback-порту, чтобы вы поставили прокси сами. Каждый режим использует собственный pbkdf2-логин панели, поэтому вкладка **Account** работает везде.

Пример автоматической установки (конфиг через переменные окружения):

```
sudo -E ROLE=master TLS_MODE=cloudflare CF_TOKEN=… PANEL_DOMAIN=panel.example.net \
     BASIC_USER=admin BASIC_PASS='…' bash -s host \
     < <(curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh)
```

## Добавление узла

Узлы управляются целиком из интерфейса — установщик больше о них не спрашивает.

1. **Nodes → Add node** — задайте имя, публичный IP/хост, который набирают клиенты, и цвет. Вы получите **одноразовый токен подключения** и точный однострочник.
2. **На входном сервере** вставьте этот однострочник — он скачает репозиторий и запустит установщик узла:
   ```
   curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh \
     | sudo bash -s node -key SECURE_NODE_KEY -host https://panel.example.net
   ```
   Он проведёт **Настройку узла** — интерфейсы (у каждого свой endpoint IP) и опциональный turn-proxy — и запустит `swg-noded`. Для Docker панель показывает и команду `… bash -s docker node -key … -host …`.
3. Через несколько секунд узел станет **online** в разделе Nodes.

С самоподписанным сертификатом панели узел по умолчанию его не проверяет — учётные данные это токен, а канал всё равно зашифрован. Чтобы проверять, используйте настоящий сертификат и ответьте «да» на вопрос о TLS, либо закрепите самоподписанный через `TLS_FINGERPRINT=<sha256-hex>` (проверяется во время рукопожатия, до отправки токена).

**Действия над узлом:** **Edit** (адрес/цвет — адрес попадает в конфиги клиентов), **Rotate token** (старый немедленно перестаёт работать; подключите узел заново), **Remove** (отзывает токен и снимает узел с пиров).

## Управление пирами

**Пир** — это одна личность (пара ключей + IP + PSK), которую можно развернуть сразу на нескольких узлах для резервирования: клиент переключается между ними, меняя `Endpoint`.

- **Add peer** — выберите интерфейс и один или несколько узлов, при желании имя. Панель выделяет свободный IP; браузер генерирует пару ключей и PSK и собирает конфиг(и). Конфиг + QR показываются **один раз** — приватный ключ не сохраняется.
- **Назначение / снятие** — развёртывание той же личности на новых узлах просто добавляет туда её публичный ключ и PSK; в конфиге клиента появляется ещё один endpoint для переключения.
- **Копировать на другой сервер** — со страницы пира скопируйте его на сервер, где его ещё нет. Используются тот же ключ и PSK, но выдаётся **новый IP** из подсети того сервера (отдельный туннель с общей личностью). Нужен приватный ключ клиента, поэтому работает при включённом `store_configs` или сразу после создания пира.
- **Remove** — убирает пир из реестра; каждый узел, где он был, удалит его при следующей синхронизации.

Статус (online, partial, dangling, …) вычисляется при каждом обновлении по снимкам узлов — пир остаётся «online», пока одна из реплик ненадолго недоступна. Для каждого размещения также показывается **транспорт** — **direct** или **via turn-proxy** (определяется, когда наблюдаемый endpoint клиента совпадает с turn-proxy узла) — и, если turn-proxy есть, его endpoint + **wrap-ключ** для настройки клиентского приложения vk-turn-proxy.

**Account** — имя пользователя и пароль панели меняются в разделе **Account**; изменение вступает в силу сразу (вас попросят войти заново).

## Docker

**Однострочники** — установят Docker при необходимости, разместят `docker-compose.yml` + `.env` в `/opt/swg-panel-docker`, поднимут профиль и **спросят всё необходимое**. Те же две точки входа, что и на bare-metal:

**Панель** — Шаг 1 спрашивает роль, как на bare-metal: **master** (панель + этот сервер сам поднимает WG/AWG в локальном контейнере-узле, регистрируется за один проход) или **host** (только панель). По умолчанию — `master`.

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s docker host
```

**Узел** — отдельный входной сервер; спросит URL панели + ключ из **Nodes → Add node** (так разворачиваются узлы при роли `host`)

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s docker node
```

Флаги пропускают вопросы (их использует команда подключения из панели): `-role master|host`, `-pass`, `-domain`, `-key`, `-host`, `-endpoint`, `-base`, `-port`, `-tls`, `-ifaces`, `-net host|bridge` — например `… bash -s docker node -key NODE_KEY -host https://panel.example.net`.

**Или вручную** — один compose-файл, три профиля с именами ролей:

**Панель + локальный узел** (роль `master`)

```
docker compose --profile master up -d
```

**Только панель** (роль `host`)

```
docker compose --profile host up -d
```

**Только входной сервер** (роль `node`)

```
docker compose --profile node up -d
```

Вручную профиль `master` **не** регистрирует локальный узел автоматически — добавьте его в **Nodes → Add node**, задайте `NODE_TOKEN` и укажите имя сервиса панели (`PANEL_URL=https://swg-panel:8443`). Роль установщика `master` делает всё это за вас. (`host-node` по-прежнему работает как псевдоним `master` для старых установок.)

Настройка через `.env` (скопируйте из `.env.example`):

- **Панель:** `PANEL_PASSWORD` (обязательно), `PANEL_USER`, `PANEL_DOMAIN`, `PANEL_BASE` (необязательный подпуть, например `/swg`), `PANEL_PORT` и `TLS` — `letsencrypt` · `cloudflare` · `cf15` · `selfsigned` · `none`, выпускается внутри контейнера встроенным `acme.sh` так же, как на bare-metal (задайте `ACME_EMAIL` / `CF_TOKEN` / `CF_ORIGIN_TOKEN` под выбранный режим; см. [TLS](#установка-панели)).
- **Узел:** `PANEL_URL` (для `master` на bridge используйте `https://swg-panel:8443`; при host-сети — loopback), `NODE_TOKEN` (из раздела Nodes), `NODE_ENDPOINT`, `NODE_IFACE` / `NODE_IFACES`, `NODE_LISTEN_PORT`, `NODE_ADDRESS`, `NODE_NET` (`host` · `bridge`), `TLS_VERIFY`, `DNS`.

**Сеть** — по умолчанию установщик ставит узел в режим **`network_mode: host`** (`NODE_NET=host`), поэтому любой UDP-порт интерфейса — **в том числе интерфейсов, созданных позже из панели** — доступен без проброса портов, а пропускная способность выше (нет `docker-proxy`). На `master` узел достаёт совмещённую панель через loopback. Выберите **`bridge`**, чтобы изолировать сетевое пространство узла; тогда `ListenPort` каждого интерфейса нужно **публиковать** в `docker-compose.yml` (панель подсказывает это и печатает точную строку при создании интерфейса на bridge-узле). Узел сообщает свой режим панели, поэтому подсказка показывается только там, где она нужна.

**Два образа** (по умолчанию тянутся готовыми из GHCR; `--build` собирает локально). `swg-panel` — чистый Python + встроенный `acme.sh`. `swg-node` несёт userspace-датаплейн **`amneziawg-go`** и инструменты `awg` (контейнер не может загрузить модуль ядра хоста) и требует `NET_ADMIN` + `/dev/net/tun`. По умолчанию управляет одним интерфейсом (AmneziaWG 2.0; `NODE_PLAIN_WG=yes` для обычного WG), несколькими через `NODE_IFACES` (`name:port:addr[:proto[:endpoint]],…`) или любыми конфигами, смонтированными в `/etc/swg-node/*.conf`. Интерфейсы, созданные из панели, сохраняются между `up -d` (каталог конфигов — это том). Masquerade добавляется автоматически. Для максимальной пропускной способности предпочтительнее **bare-metal**-узел с модулем ядра. При включённом управлении turn-proxy `swg-noded` запускает каждый прокси как соседний контейнер `swg-turn-*` (host-сеть, `--restart unless-stopped`) через смонтированный сокет Docker — правка такого прокси просто пересоздаёт контейнер, без systemd на хосте.

**Повторный запуск установщика безопасен** — он обнаруживает существующую установку, сохраняет `.env` + `./data` (токен, логин, сертификат, интерфейсы) и предлагает текущие значения по умолчанию; шаги wg/awg и turn-proxy показывают, что уже есть, и позволяют добавить ещё. Чтобы начать с нуля, сначала запустите деинсталлятор. (Docker: применяйте через `PULL_POLICY=always … bootstrap.sh docker host|node`, чтобы подтянулись новый образ и compose.)

## Конвертация между bare-metal и Docker

Сменить **метод** установки на месте: перезапустите установщик, запросив *другой* метод — он предложит **конвертировать · оставить · прервать**.

```bash
# тот же one-liner, но с другим методом — например, Docker-master ⇆ bare-metal:
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s master         # → bare-metal (голое слово роли = bare-metal)
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s docker master  # → Docker
```

Конвертация **сохраняет всё** — URL панели, логин, реестр, узлы и TLS-сертификат, а для `master` — токен локального узла, его интерфейсы и turn-proxy. Принцип **copy-first**: новый метод полностью разворачивается *до* сноса старого, поэтому простой — лишь несколько секунд атомарного переключения, а узлы самовосстанавливаются на следующей синхронизации.

**Роли конвертируются независимо.** При конвертации `master` сначала спрашивается, переносить ли всё целиком или только одну половину: **весь master** (панель + узел, по умолчанию), **только host** (только панель) или **только node** (только узел). Сплит оставляет **сервер со смешанными методами** (например, Docker-панель с bare-metal-узлом), которым вы управляете в двух местах — удобно, чтобы перевести узел на датаплейн ядра, оставив панель на Docker.

**Интерфейсы и turn-proxy переносятся на шаге узла.** По мере конвертации узла он перечисляет управляемые им интерфейсы wg/awg и turn-proxy и для каждого набора спрашивает **«Перенести эти … в … узел? (Y/n)»**; ключи, порты и endpoint'ы переносятся (copy-first), а оригиналы продолжают работать до переключения. Откажитесь — и они останутся на старом методе, а новый узел стартует пустым. Оба направления работают одинаково.

**Статус в реальном времени.** Заголовок панели — и плитка узла, для node или master — показывает **конвертация → сконвертировано** по ходу процесса (метка *конвертация* в момент подтверждения, *сконвертировано* — вместе с финальной сводкой), так что за конвертацией можно следить из консоли.

> Для узлов предпочтительнее **docker → bare-metal**: модуль ядра хоста быстрее, чем userspace-`amneziawg-go` в контейнере. Узел, отделённый от совмещённого Docker-master, автоматически направляется на loopback-порт панели, так как DNS-имя compose недоступно вне сети контейнеров.

## Справочник по конфигурации

**Панель — `/etc/swg-panel/fleet.json`**:

```json
{
  "roster_path":   "/var/lib/swg-panel/users.json",
  "nodes_path":    "/var/lib/swg-panel/nodes.json",
  "stats_dir":     "/var/www/wgstats",
  "store_configs": false,
  "config_dir":    "/var/lib/swg-panel/configs",
  "node_interval": 5
}
```

`store_configs` хранит сгенерированные конфиги клиентов (включая приватные ключи) на панели — по умолчанию выключено; не включайте без понимания последствий.

**Узел — `/etc/swg-agent/config.json`**:

```json
{
  "interfaces": { "awg0": { "cmd": ["awg"], "conf": "/etc/amnezia/amneziawg/awg0.conf" } },
  "endpoint_host": "203.0.113.7",
  "dns": ["1.1.1.1"],
  "panel": { "url": "https://panel.example.net", "token": "…", "verify": false },
  "node":  { "interval": 5, "agent": "/opt/swg-agent/swg-agent", "sudo": false }
}
```

Узел может обслуживать несколько интерфейсов — перечислите их все в `interfaces`; каждый пир применяется к тому, которому принадлежит. Поставьте `panel.verify: true` для настоящего сертификата или добавьте `panel.fingerprint: "<sha256-hex>"`, чтобы закрепить самоподписанный.

## Эксплуатация

- **Логи:** `journalctl -u swg-panel-server -f` (панель), `journalctl -u swg-noded -f` (узел). Docker: `docker compose logs -f`.
- **Сменить токен узла:** Nodes → ⋯ → Rotate token, затем повторите команду установки (или обновите `config.json` узла).
- **Удалить узел:** Nodes → Remove. Чтобы вывести машину из строя, остановите на ней `swg-noded`.
- **Резервная копия:** `users.json` + `nodes.json` (в `/var/lib/swg-panel`) — это всё состояние. Скопируйте их в надёжное место.
- **Обновление:** `… | sudo bash -s update` (или `./update.sh`). Тянет свежий код, сам определяет установленное (bare-metal панель/узел и/или Docker), обновляет бинарники/SPA и перезапускает — конфиг и состояние сохраняются. Версия пишется в файл `VERSION` каждого компонента.
- **Удаление:** `… | sudo bash -s uninstall` (или `./uninstall.sh`). Перечисляет все установленные компоненты — панель, узел на «голом железе», развёртывание Docker, AmneziaWG, WireGuard и **каждый** сервер turn-proxy — затем по очереди спрашивает, удалить или оставить каждый. Ничего не удаляется без подтверждения; можно сохранить реестр / список узлов / каталог данных Docker для переустановки. `--yes` удаляет всё, `--dry-run` показывает план.

## Безопасность

- **Транспорт:** узлы всегда подключаются только *наружу* к панели по TLS. Предпочтителен настоящий сертификат (`letsencrypt`/`cloudflare`), чтобы узлы работали с `verify: true`; с самоподписанным — закрепляйте отпечаток.
- **Токены узлов** аутентифицируют узел перед панелью. Показываются один раз, хранятся только в виде хеша, могут быть сменены. Считайте `config.json` узла (где лежит «живой» токен) секретом — он в режиме `600`.
- **PSK** генерируются для каждого пира и хранятся в реестре, чтобы все узлы оставались согласованными; держите `/var/lib/swg-panel` доступным на чтение только пользователю панели.
- **Приватные ключи** генерируются в браузере и не передаются на сервер (если вы намеренно не включили `store_configs`). Если ключ пира утерян — выпустите пир заново.

## Диагностика

- **Узел остаётся «awaiting enroll» / не подключается.** Он ещё не синхронизировался. Смотрите `journalctl -u swg-noded -f` на узле: `fingerprint mismatch` — неверный отпечаток; HTTP `401` — неверный или сменённый токен; ошибка соединения — не тот URL/порт панели или закрыт фаервол.
- **Узел «offline».** Он синхронизировался раньше, но замолчал — проверьте демон и исходящую сеть узла.
- **Пиры не появляются на узле.** Убедитесь, что пир назначен на этот узел и его интерфейс совпадает с тем, что узел действительно обслуживает. Данные берутся из последнего снимка узла, поэтому только что добавленному узлу нужна одна синхронизация.
- **Браузер предупреждает о сертификате.** Это нормально при `selfsigned`. Для продакшена используйте настоящий сертификат или примите предупреждение за доверенным туннелем.
- **Клиент подключается (рукопожатие проходит), но нет интернета / `rx` растёт, а `tx` ≈ 0.** Узел расшифровывает пакеты, но не маршрутизирует их наружу. Интерфейсы, созданные установщиком, настраивают это автоматически; для интерфейса, сделанного вручную, включите форвардинг (`net.ipv4.ip_forward=1`) и добавьте masquerade для подсети туннеля через WAN-интерфейс (`iptables -t nat -A POSTROUTING -s <subnet> -o <wan> -j MASQUERADE`).
- **Умная маршрутизация переключается не мгновенно — подождите минуту-две.** Включение списка/категории или смена его выхода применяются с задержкой (*eventually consistent*), не сразу. Узел применяет маршрутизацию в периодической сверке (~60 c) после того, как подтянет с панели готовый список, — поэтому проходит некоторое время, прежде чем подходящий трафик начнёт уходить через новый выход; а если соединение уже открыто, оно продолжает идти по-старому до переподключения. Присвоение категории отстаёт ещё немного: IP-набор назначения наполняется лениво (по мере того как его домены резолвятся/наблюдаются), поэтому только что перенаправленный трафик какое-то время может показываться как **некатегоризированный**, прежде чем попадёт в свою категорию. Всё сходится само — ничего не сломано; повторная проверка через минуту или переподключение клиента ускоряют процесс.
- **`acme.sh` не смог выпустить сертификат.** Для `letsencrypt` нужен доступный порт 80; если 80 занят — используйте `cloudflare` (DNS-01, без порта 80).
