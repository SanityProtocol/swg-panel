# swg-panel

Самостоятельно разворачиваемая панель управления для небольшого сервиса WireGuard / AmneziaWG, работающего на одном или нескольких входных серверах. Панель — единственный источник истины о ваших пирах; каждый входной сервер (**узел**) синхронизируется с ней по **исходящему HTTPS** и приводит себя к назначенному ему набору пиров — поэтому узлам не нужен ни входящий доступ, ни SSH, ни rsync.

Продумана для multi-hop (входной узел на границе → удалённый выходной узел) и рассчитана на масштабирование — от пары пиров на одной машине до крупных флотов на множестве серверов.

## Как это работает

- **Панель** (control plane) — отдаёт веб-интерфейс, обслуживает собственный TLS + логин и API синхронизации узлов. Ей принадлежат ростер (ваши пиры) и хранилище узлов. Чистый Python, без базы данных.
- **Узел** (входной сервер) — запускает `swg-noded`, который каждые несколько секунд отправляет панели снимок состояния своего интерфейса, получает набор пиров, которые у него *должны* быть, и приводит интерфейс к нужному состоянию локально через `swg-agent` (добавляя/удаляя пиров на живом интерфейсе wg/awg). Только исходящий HTTPS.
- **Декларативность** — вы меняете пиров в панели, узлы сходятся к нужному состоянию. Узел, пропустивший такт, самовосстанавливается при следующей синхронизации, а кратковременный сбой панели никогда не сотрёт пиров узла (узел применяет изменения только по корректному ответу).
- **Ключи генерируются у вас в браузере** — пары ключей пиров всегда создаются на стороне клиента. **По умолчанию** (`store_configs` включён) панель хранит копию каждого сгенерированного конфига — включая приватный ключ — чтобы его QR/скачивание оставались доступными в любой момент. Не хотите держать секреты на диске? Выключите `store_configs` (Settings → Client configs), и панель сохранит только публичный ключ, назначенный IP и preshared-ключ; приватный ключ показывается **один раз** при создании. PSK всегда принадлежит панели, поэтому каждый узел, где живёт пир, остаётся согласованным.

## Метод × Роль

Два независимых выбора.

**Метод** — как устанавливается:
- **bare-metal** — systemd-сервисы, через `install-host.sh` / `install-node.sh`.
- **docker** — `docker compose` с образами `swg-panel` / `swg-node`.

**Роль** — что делает машина:
- **host** — только панель.
- **master** — панель *и* при этом машина сама является входным сервером (заводится в панель автоматически).
- **node** — только входной сервер.

Комбинируйте свободно: панель в Docker с bare-metal-узлами, bare-metal `master` плюс дополнительные Docker-узлы и так далее.

## Компоненты

| компонент | где | что делает |
|---|---|---|
| `swg-panel-server` | host | UI + API синхронизации узлов + хранилище ростера/узлов; обслуживает собственный TLS + логин и статусную панель |
| `swg-noded` | каждый узел | отправляет снимки, забирает желаемый набор пиров, приводит интерфейс к нему локально — только исходящий HTTPS |
| `swg-agent` | каждый узел | применяет одно добавление/удаление пира на живом интерфейсе и в его `.conf` |
| `users.json` | host | **ростер** — ваши пиры (публичный ключ, IP, PSK и на каких узлах каждый живёт) |
| `nodes.json` | host | **хранилище узлов** — имена узлов, endpoint'ы и хеши токенов enrollment |

Вы никогда не редактируете `users.json` или `nodes.json` вручную — это делает UI.

## Быстрый старт

Четыре однострочника — каждый **спрашивает всё, что ему нужно**. Выберите метод (bare-metal или Docker) для каждой машины; комбинируйте свободно.

### A — bare-metal (systemd)

**Панель** — спрашивает роль: master (панель + эта машина является узлом) или host (только панель)

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s host
```

**Узел** — спрашивает URL панели + ключ из Nodes → Add node

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s node
```

### B — Docker

**Панель** — при необходимости устанавливает Docker, затем спрашивает роль (master: панель + эта машина является узлом, заводится автоматически · host: только панель) и домен + выбор TLS (логин генерируется автоматически, поменяете его позже в панели)

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s docker host
```

**Узел** — спрашивает URL панели + ключ из Nodes → Add node

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s docker node
```

Откройте URL панели, войдите и добавьте входные серверы через **Nodes → Add node** — для каждого печатается точная команда и для bare-metal, *и* для Docker (ключ уже подставлен). Подробнее: [Установка панели](#установка-панели) · [Добавление узла](#добавление-узла) · [Docker](#docker).

**Обновить** любую машину позже, на месте (автоматически определяет, что установлено):

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s update
```

## Установка панели

`install-host.sh` (запускается из `bootstrap.sh host` или напрямую из клона) настраивает панель и, для роли `master`, локальный входной сервер. Две последовательные части — **настройка панели**, затем **настройка узла** (только для master):

| часть | запрос | значение |
|---|---|---|
| Panel · 1 | **Роль** | `master` (по умолчанию — панель + эта машина является входным сервером) или `host` (только панель) |
| Panel · 2 | **URL панели** | IP, host или host с подпутём (`vpn.example.com/swg`), чтобы смонтировать панель под существующим сайтом. По умолчанию — публичный IP этой машины. |
| Panel · 3 | **TLS** | `letsencrypt` (по умолчанию) · `cloudflare` · `cf15` · `selfsigned` · `skip` |
| Panel · 4 | **Режим отдачи** | `internal` (по умолчанию, самодостаточный) · `nginx` · `caddy` · `skip` |
| Panel | **Порт / админ** | порт (по умолчанию **443** для `internal`; unit добавляет capability для привязки к портам < 1024) и веб-логин — предлагает `admin` + 3 цифры; меняется позже в разделе **Account** |
| Node · 1 | **Имя узла** | *(master)* имя узла этой машины (по умолчанию: hostname) |
| Node · 2 | **Endpoint IP** | *(master)* публичный IP, на который клиенты подключаются к этой машине (по умолчанию: определённый) |
| Node · 3 | **Интерфейсы** | *(master)* управляет **всеми** найденными интерфейсами wg/awg (сканирует всё в `/etc/amnezia` и `/etc/wireguard`); предлагает **создать** интерфейс, если их нет; нажмите **Enter**, чтобы продолжить, или **`new`**, чтобы добавить ещё один. Новые интерфейсы получают серверный ключ, туннельную подсеть и автоматический forward + masquerade (`PostUp`/`PostDown`), чтобы клиенты выходили в интернет. |
| Node · 4 | **Turn-proxy** | *(master)* опциональный [vk-turn-proxy](https://github.com/cacggghp/vk-turn-proxy) — туннелирует wg/awg через TURN-серверы VK/Yandex. Определяет любой установленный прокси (по его unit'у с `-listen`/`-connect`) и показывает его; **Enter** — пропустить, либо выберите форк по **номеру**, чтобы установить (`WINGS-N`/`samosvalishe`/`kiper292`/`Moroka8` для Android, `anton48` для iOS). Он скачивает release-бинарник, **автоматически выводит** порт `-connect` из только что настроенного интерфейса wg/awg (перечисляет все), **генерирует 64-hex wrap-ключ** с флагами форка (`-wrap-srtp`/`-wrap`/`-wrap-mode`; у `kiper292` их нет) и записывает `listen`/`connect`/`wrap_key` для конфигов панели + клиентов. На **bare-metal** каждый прокси — systemd-сервис, чья цель хранится в `EnvironmentFile`, поэтому правка в панели просто переписывает его + перезапускает (без `daemon-reload`); в **Docker** это отдельный **контейнер**-сосед (`swg-turn-*`, управляемый через смонтированный сокет Docker — без `--privileged`/`--pid=host`). |

**TLS:**
- **letsencrypt** (по умолчанию) — настоящий сертификат через `acme.sh` (HTTP-01 standalone для `internal`/`caddy`, webroot за `nginx`); нужен доступный порт 80.
- **cloudflare** — настоящий сертификат через DNS-01; **порт 80 не используется вообще**. Токену нужны права `Zone:DNS:Edit` + `Zone:Read`.
- **cf15** — **Origin**-сертификат Cloudflare, **15 лет**, выпускается через CF API (нужен **API-токен** с правами `Zone` → `SSL and Certificates` → `Edit`; устаревший Origin CA Key более не используется). ⚠️ Доверенный только **за прокси Cloudflare** (оранжевое облако) — при прямом обращении к origin браузер покажет недоверенный сертификат. Продление не требуется 15 лет.
- **selfsigned** — мгновенно; браузеры предупредят один раз. Годится для старта или за туннелем.
- **skip** — принесите свой сертификат (задайте `CERT_FULLCHAIN`/`CERT_KEY`) или терминируйте TLS в другом месте. Без сертификата панель/прокси отдаёт обычный HTTP.

**Режимы отдачи:** `internal` — самодостаточный (собственный TLS + логин); `nginx` / `caddy` — reverse-proxy на loopback (TLS терминирует прокси, с блоком `location`/`handle`, учитывающим любой подпуть); `skip` оставляет панель на loopback-порту, чтобы вы поставили фронт сами. Все режимы используют собственный pbkdf2-логин панели, поэтому вкладка **Account** работает везде.

Пример неинтерактивной установки (конфигурация через env):

```
sudo -E ROLE=master TLS_MODE=cloudflare CF_TOKEN=… PANEL_DOMAIN=panel.example.net \
     BASIC_USER=admin BASIC_PASS='…' bash -s host \
     < <(curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh)
```

## Добавление узла

Узлы управляются целиком из UI — установщик больше о них не спрашивает.

1. **Nodes → Add node** — задайте имя, публичный IP/host, на который подключаются клиенты, и цвет. Вы получите **одноразовый токен enrollment** и точный однострочник.
2. **На входном сервере** вставьте этот однострочник — он скачивает репозиторий и запускает установщик узла:
   ```
   curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh \
     | sudo bash -s node -key SECURE_NODE_KEY -host https://panel.example.net
   ```
   Он выполняет **настройку узла** — интерфейсы (каждый со своим endpoint IP) и опциональный turn-proxy — и запускает `swg-noded`. Предпочитаете Docker? Панель показывает и команду `… bash -s docker node -key … -host …`.
3. Через несколько секунд узел становится **online** на экране Nodes.

Для **самоподписанной** панели установщик **автоматически пинит сертификат при первом контакте** (trust-on-first-use): узел сохраняет его sha256 и проверяет при каждом handshake, до отправки токена — так что MITM не сможет выдать себя за панель даже без CA. Панель с **настоящим CA** проверяется по системному хранилищу доверия. Переопределить: `TLS_VERIFY=yes|no` или явный `TLS_FINGERPRINT=<sha256-hex>`. Если панель позже сменит адрес (host/port), узел **автоматически перенаправится** на новый — но только если тот предъявляет запиненный/доверенный сертификат.

**Действия над узлом:** **Edit** (endpoint/цвет — endpoint попадает в клиентские конфиги), **Rotate token** (старый перестаёт работать немедленно; заведите узел заново), **Remove** (отзывает токен и снимает назначение узла).

## Управление пирами

**Пир** — это одна идентичность (пара ключей + IP + PSK), которую можно развернуть сразу на нескольких узлах для отказоустойчивости — клиент переключается между ними, меняя свой `Endpoint`.

- **Add peer** — выберите интерфейс и один или несколько узлов, при желании имя. Панель выделяет свободный IP; ваш браузер генерирует пару ключей и PSK и собирает конфиг(и). Вам **один раз** показываются конфиг + QR — приватный ключ не сохраняется. QR крупный, по тапу разворачивается на весь экран для удобного сканирования с телефона.
- **Assign / unassign** — развёртывание той же идентичности на большем числе узлов просто добавляет туда её публичный ключ + PSK; клиентский конфиг получает ещё один endpoint для переключения.
- **Copy to another server** — со страницы пира скопируйте его на сервер, где его ещё нет. Он переиспользует тот же ключ + PSK, но получает **новый IP** из подсети этого сервера (отдельный туннель, разделяющий идентичность). Требует приватного ключа клиента, поэтому работает при включённом `store_configs` или сразу после создания пира.
- **Remove** — убирает пира из ростера; каждый узел, где он жил, удалит его при следующей синхронизации.

Живой статус (online, partial, dangling, …) вычисляется при каждом обновлении из снимков узлов — пир остаётся «online», пока одна из реплик кратковременно недоступна. Для каждого развёртывания также показывается его **транспорт** — **direct** или **via turn-proxy** (определяется, когда наблюдаемый endpoint клиента совпадает с turn-proxy узла) — и, где есть turn-proxy, endpoint прокси + **wrap-ключ** для настройки клиентского приложения vk-turn-proxy.

**Account** — сменить имя пользователя/пароль панели можно на вкладке **Account**; изменение вступает в силу немедленно (вам предложат войти заново).

## Подписки и контроль доступа

**Подписки (`swg-sub`)** — отдельная, публичная, **read-only** поверхность, которая отдаёт каждому пользователю личную страницу по адресу `https://sub.<domain>/<token>#<unlock-key>` с его конфигом и QR для **каждого** узла, где он есть: **WireGuard**, **AmneziaWG** и каждый назначенный ему **TURN-прокси-форк** (WINGS-N, samosvalishe, Moroka8, cacggghp, …), плюс поле VK-call-ссылки для FreeTurn, бейджи протокола/реле, светлая/тёмная тема, RU/EN и копирование / скачивание / отправка. **Выключено по умолчанию** — включается в **Settings → Subscriptions**.

- **Хранение ключей.** Персональный **ключ расшифровки живёт в `#fragment` ссылки** — никогда не отправляется на сервер. Панель хранит только **шифротекст** (зашифрованные блобы конфигов) плюс публичную карту токенов (`subs/users.json` = `{token_sha}`); обёрнутые SK ключи расшифровки лежат в отдельном `subs/escrow.json`, который эта поверхность прочитать не может. Компрометация публичной страницы даёт только шифротекст, никогда — приватный ключ (поэтому от хранения `store_configs` в открытом виде отказались).
- **Изоляция.** `swg-sub` — отдельный процесс/контейнер под выделенным малопривилегированным пользователем: монтирует состояние панели `:ro` и **маскирует** каждый секрет, который не должен открывать (`auth`, `panel-settings.json`, `subs/vault.json`, `subs/escrow.json`, приватный ключ TLS) через `/dev/null` + `tmpfs`. Никакого кода логина, записи или узла.
- **Отдача.** Конфиг зеркалится в `subs/serve.json` (`enabled`, `serve.{host,port,tls_mode,cert_path,key_path}`, языки). `swg-sub` терминирует собственный TLS — свой сертификат Let's Encrypt для `sub.<domain>`, явный путь к сертификату или режим `reverse-proxy` (простой HTTP за вашим прокси). Docker: контейнер `swg-sub` (`:8444`, поставьте фронт — reverse-proxy / Cloudflare). Bare-metal: systemd-сервис `swg-sub`.

**Приостановка / блокировка доступа** — декларативный флаг `disabled` на пользователе (или на конкретном узле). Установите его — и следующая сверка узла **удалит пира** (туннели останавливаются) *и* **приостановит подписку** (страница гаснет), но `token_sha` и ключи сохраняются, поэтому разблокировка мгновенна: пир добавляется обратно, страница восстанавливается **без перевыпуска ключей**.

## Docker

**Однострочники** — при необходимости устанавливают Docker, разворачивают `docker-compose.yml` + `.env` в `/opt/swg-panel-docker`, поднимают профиль и **спрашивают всё нужное**. Те же две точки входа, что и на bare-metal:

**Панель** — Шаг 1 спрашивает роль, ровно как на bare-metal: **master** (панель + эта машина также запускает WG/AWG как совмещённый контейнер-узел, заводится в один проход) или **host** (только панель). По умолчанию — `master`.

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s docker host
```

**Узел** — отдельный входной сервер; спрашивает URL панели + ключ из **Nodes → Add node** (именно так разворачиваются узлы для роли `host`)

```
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s docker node
```

Флаги пропускают запросы (команда enroll из панели их использует): `-role master|host`, `-pass`, `-domain`, `-key`, `-host`, `-endpoint`, `-base`, `-port`, `-tls`, `-ifaces`, `-net host|bridge` — например `… bash -s docker node -key NODE_KEY -host https://panel.example.net`.

**Или вручную** — один compose-файл, три профиля по именам ролей:

**Панель + локальный узел** (роль `master`)

```
docker compose --profile master up -d
```

**Только панель** (роль `host`)

```
docker compose --profile host up -d
```

**Только входной сервер** (роль `node`)

```
docker compose --profile node up -d
```

Вручную профиль `master` **не** заводит локальный узел автоматически — добавьте его в **Nodes → Add node**, задайте `NODE_TOKEN` и направьте его на service name панели (`PANEL_URL=https://swg-panel:8443`). *Роль установщика* `master` делает всё это за вас. (`host-node` всё ещё работает как алиас `master` для старых установок.)

Конфигурация через `.env`:

- **Панель:** `PANEL_PASSWORD` (обязателен), `PANEL_USER`, `PANEL_DOMAIN`, `PANEL_BASE` (опциональный подпуть, напр. `/swg`), `PANEL_PORT` и `TLS` — `letsencrypt` · `cloudflare` · `cf15` · `selfsigned` · `none`, выпускается внутри контейнера встроенным `acme.sh` ровно как на bare-metal (задайте `ACME_EMAIL` / `CF_TOKEN` / `CF_ORIGIN_TOKEN` в зависимости от выбранного режима; см. [TLS](#установка-панели)).
- **Узел:** `PANEL_URL` (для `master` в bridge — `https://swg-panel:8443`; host networking использует loopback), `NODE_TOKEN` (с экрана Nodes), `NODE_ENDPOINT`, `NODE_IFACE` / `NODE_IFACES`, `NODE_LISTEN_PORT`, `NODE_ADDRESS`, `NODE_NET` (`host` · `bridge`), `TLS_VERIFY`, `DNS`.

**Сеть** — установщик по умолчанию помещает узел в **`network_mode: host`** (`NODE_NET=host`), поэтому UDP-порт каждого интерфейса — **включая интерфейсы, которые вы позже создадите из панели** — доступен без покортового маппинга, а пропускная способность выше (нет `docker-proxy`). На `master` узел достаёт до совмещённой панели через loopback. Выберите **`bridge`**, чтобы, наоборот, изолировать сетевое пространство имён узла; тогда `ListenPort` каждого интерфейса нужно **опубликовать** в `docker-compose.yml` (панель об этом предупреждает и печатает точную строку, когда вы создаёте интерфейс на bridge-узле). Узел сообщает панели свой режим, поэтому подсказка показывается только там, где нужна.

**Два образа** (по умолчанию тянутся готовыми из GHCR; `--build` собирает их локально). `swg-panel` — чистый Python + встроенный `acme.sh`. `swg-node` несёт userspace-датаплейн **`amneziawg-go`** + инструменты `awg` (контейнер не может загрузить модуль ядра хоста) и требует `NET_ADMIN` + `/dev/net/tun`. По умолчанию он управляет одним интерфейсом (AmneziaWG 2.0; `NODE_PLAIN_WG=yes` для обычного WG), несколькими через `NODE_IFACES` (`name:port:addr[:proto[:endpoint]],…`) или любыми конфигами, которые вы смонтируете в `/etc/swg-node/*.conf`. Созданные из панели интерфейсы переживают `up -d` (каталог конфигов — это volume). Masquerade настраивается автоматически. Для лучшей пропускной способности предпочтителен **bare-metal**-узел с модулем ядра. При включённом управлении turn-proxy `swg-noded` запускает каждый прокси как контейнер-сосед `swg-turn-*` (host network, `--restart unless-stopped`) через смонтированный сокет Docker — правка одного просто пересоздаёт контейнер, без systemd на хосте.

Готовые образы: `ghcr.io/sanityprotocol/swg-panel` и `ghcr.io/sanityprotocol/swg-node`.

**Повторный запуск установщика безопасен** — он обнаруживает существующую установку, сохраняет ваши `.env` + `./data` (токен, логин, сертификат, интерфейсы) и предлагает текущие значения по умолчанию; шаги wg/awg и turn-proxy показывают, что уже есть, и позволяют добавить ещё. Чтобы начать с нуля, сначала запустите деинсталлятор. (Docker: примените заново с `PULL_POLICY=always … bootstrap.sh docker host|node`, чтобы приехали новый образ + compose.)

## Переключение между bare-metal и Docker

Смените **метод** машины на месте: перезапустите установщик, запросив *другой* метод, и он предложит **convert · keep · abort**.

```bash
# тот же однострочник с другим методом — напр. Docker master ⇆ bare-metal:
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s master         # → bare-metal (голое слово-роль означает bare-metal)
curl -fsSL https://raw.githubusercontent.com/SanityProtocol/swg-panel/main/bootstrap.sh | sudo bash -s docker master  # → Docker
```

Конверсия **сохраняет всё** — URL панели, логин, ростер, узлы и TLS-сертификат, а для `master` — токен, интерфейсы и turn-proxy локального узла. Она работает по принципу **copy-first**: новый метод полностью разворачивается *до* сноса старого, поэтому единственный простой — несколько секунд атомарного переключения, а узлы самовосстанавливаются при следующей синхронизации.

**Роли конвертируются независимо.** При конверсии `master` сначала спрашивается, переносить ли всю машину или только половину — **весь master** (панель + узел, по умолчанию), **только host** (только панель) или **только node** (только узел). Разделение оставляет **машину со смешанным методом** (напр. панель в Docker с bare-metal-узлом), которой вы управляете в двух местах — удобно, чтобы перевести узел на датаплейн ядра, оставив панель в Docker.

**Интерфейсы и turn-proxy переносятся на этапе узла.** По мере конверсии узел перечисляет управляемые им интерфейсы wg/awg и turn-proxy-серверы и для каждого набора спрашивает **«Transfer these … into the … node? (Y/n)»**; ключи, порты и endpoint'ы переносятся (copy-first), а оригиналы продолжают обслуживать до переключения. Откажитесь, чтобы оставить их на старом методе и запустить новый узел пустым. Оба направления ведут себя одинаково.

**Живой статус.** Заголовок панели — и плитка узла, для узла или master — показывает **converting → converted** по ходу процесса (тег *converting* — в момент подтверждения, *converted* — с итоговой сводкой), так что за конверсией можно наблюдать из консоли.

> Для узлов предпочтителен переход **docker → bare-metal** — модуль ядра хоста производительнее userspace-`amneziawg-go` в контейнере. Узел, отделённый от совмещённого Docker-master, автоматически направляется на loopback-порт панели, поскольку DNS-имя из compose недоступно снаружи сети контейнера.

## Справочник по конфигурации

**Панель — `/etc/swg-panel/fleet.json`**:

```json
{
  "roster_path":   "/var/lib/swg-panel/users.json",
  "nodes_path":    "/var/lib/swg-panel/nodes.json",
  "stats_dir":     "/var/www/wgstats",
  "store_configs": true,
  "config_dir":    "/var/lib/swg-panel/configs",
  "node_interval": 5
}
```

`store_configs` хранит сгенерированные клиентские конфиги (включая приватные ключи) на панели, чтобы QR-коды и скачивания оставались доступными в любой момент — **включено по умолчанию** (установщики пишут `true`). Задайте `false` для режима **без секретов на диске**: тогда панель хранит только публичный ключ, IP и PSK, а приватный ключ каждого конфига показывается один раз при создании. Панель отмечает, когда оно выключено (существующим пирам тогда нужна разовая ротация ключа, чтобы заново получить конфиг).

**Узел — `/etc/swg-agent/config.json`**:

```json
{
  "interfaces": { "awg0": { "cmd": ["awg"], "conf": "/etc/amnezia/amneziawg/awg0.conf" } },
  "endpoint_host": "203.0.113.7",
  "dns": ["1.1.1.1"],
  "panel": { "url": "https://panel.example.net", "token": "…", "verify": false },
  "node":  { "interval": 5, "agent": "/opt/swg-agent/swg-agent", "sudo": false }
}
```

Узел может обслуживать несколько интерфейсов — перечислите их все в `interfaces`; каждый пир применяется к тому, которому принадлежит. Задайте `panel.verify: true` для настоящего сертификата или добавьте `panel.fingerprint: "<sha256-hex>"`, чтобы запинить самоподписанный.

## Эксплуатация

- **Логи:** `journalctl -u swg-panel-server -f` (панель), `journalctl -u swg-noded -f` (узел). Docker: `docker compose logs -f`.
- **Ротация токена узла:** Nodes → ⋯ → Rotate token, затем перезапустите команду установки (или обновите `config.json` узла).
- **Удаление узла:** Nodes → Remove. Остановите `swg-noded` на самой машине, чтобы вывести её из строя.
- **Резервное копирование:** `users.json` + `nodes.json` (в `/var/lib/swg-panel`) — это всё состояние: скопируйте каталог куда-нибудь за пределы машины, и вы сможете восстановить панель где угодно.
- **Автоматические резервные копии + самовосстановление:** каждая запись в критичный файл состояния (`users.json`, `nodes.json`, `panel-settings.json`) также создаёт рядом копию с меткой времени — `users.json.bak.<epoch>` — и оставляет только **последние 8**. При запуске загрузчик проверяет каждый основной файл и, если один из них повреждён/пуст (некорректное завершение, переполнение диска), **автоматически восстанавливает самую свежую исправную копию** — без вмешательства оператора. Таким образом резервная копия появляется после *каждого изменения*, а не по расписанию. (Копии за пределами машины всё равно рекомендуются на случай потери всего хоста.)
- **Обновление:** `… | sudo bash -s update` (или `./update.sh`). Забирает свежий код, автоматически определяет, что установлено (bare-metal панель/узел и/или Docker), обновляет бинарники/SPA и перезапускает — конфигурация + состояние сохраняются. Установленная версия штампуется в файл `VERSION` каждого компонента (репозиторий: [`VERSION`](VERSION)).
- **Переустановка сохраняет данные:** повторный запуск *любого* установщика на машине обнаруживает существующую установку и сохраняет всё (логин, сертификат, ростер, узлы, токен узла, интерфейсы, turn-proxy), предлагая текущие значения по умолчанию — безопасно после прерванного запуска или для изменения какой-либо опции. Чтобы начать с чистого листа, сначала выполните деинсталляцию.
- **Восстановление идентичности узла:** переустановленный или сконвертированный узел сохраняет свою идентичность после регистрации в `/var/lib/swg-recovery`, поэтому может снова присоединиться **без повторной регистрации**. Откройте это меню напрямую командой `… | sudo bash -s recovery` (работает даже на машине с уже действующим узлом — например, чтобы заново привязать оставшийся токен после пересборки).
- **Перенос панели:** смените её host/port — и **запиненные узлы автоматически перенаправятся** на следующей синхронизации, пока новый адрес предъявляет их доверенный сертификат — так что миграция панели не требует повторной регистрации на каждом узле.
- **Деинсталляция:** `… | sudo bash -s uninstall` (или `./uninstall.sh`). Перечисляет каждый установленный компонент — панель, bare-metal-узел, контейнеры панели и узла в Docker, AmneziaWG, WireGuard и **каждый** turn-proxy-сервер — затем проходит по ним и спрашивает, удалить или оставить каждый. Ничего не удаляется без «да»; можно сохранить ростер / хранилище узлов / каталог данных Docker для переустановки. `--yes` удаляет всё, `--dry-run` показывает предварительный просмотр.

## Внешний API и интеграции

**Read-only** поверхность REST + Prometheus плюс исходящие **webhook'и**, чтобы связать панель с внешним мониторингом и автоматизацией — Grafana, Uptime Kuma, Prometheus, Terraform/Ansible или вашими собственными скриптами. Она отдаёт состояние, которое панель и так собирает из синхронизаций узлов; она **никогда не меняет флот**.

- **Только чтение.** Ни один мутирующий endpoint не принимает токен — утёкший токен может наблюдать, но никогда не изменять.
- **Только панель.** Узлы не затрагиваются; они синхронизируются ровно как раньше. API отдаётся из тех же снимков в памяти + ростера, которые панель уже держит — без дополнительной нагрузки на узлы, без изменений в цикле синхронизации.
- **Дёшево.** Вне пути опроса SPA; общефлотовое представление кешируется ~3с и разделяется всеми endpoint'ами и scrape'ами, поэтому агрессивная связка Prometheus/Grafana не может деградировать панель.

Включите её в **Settings → Integrations**: выпустите токен (показывается один раз), при желании добавьте webhook'и.

**Аутентификация.** Каждому endpoint'у, кроме проб живучести, нужен токен, передаваемый одним из трёх способов:

```
Authorization: Bearer <token>
X-API-Key: <token>
?token=<token>          # query-параметр, для инструментов, которые не могут задать заголовок
```

Токены выпускаются в UI (`swgp_…`); хранится только SHA-256-хеш. Также аутентифицирует сессионная cookie администратора (чтобы страница настроек могла показывать предпросмотр). Отключение API **сразу** отклоняет все токены. `GET /healthz` и `GET /api/v1/health` — неаутентифицированные пробы живучести и не раскрывают секретов.

**Endpoint'ы:**

| endpoint | auth | что возвращает |
|---|---|---|
| `GET /healthz` | нет | текстовое `ok` для проб uptime / балансировщиков |
| `GET /api/v1/health` | нет | живучесть + грубые счётчики узлов/пиров + версия |
| `GET /api/v1/servers` | да | каждый узел со статусом + счётчиками + пропускной способностью (mesh-связи исключены) |
| `GET /api/v1/servers/{id}` | да | один сервер по **id или имени** (та же форма, что элемент `/servers`) |
| `GET /api/v1/servers/{id}/peers` | да | пиры, наблюдаемые на этом узле, с таймингом last-handshake, обогащённые данными ростера |
| `GET /api/v1/peers` | да | идентичности пиров по всему ростеру с присутствием на каждом target |
| `GET /api/v1/summary` | да | итоги по флоту + агрегированная пропускная способность |
| `GET /metrics` | да | текстовая экспозиция Prometheus (`v0.0.4`) |

**Метрики Prometheus** — **кардинальность только по узлам** (никогда по пирам, что взорвало бы число серий; детали по пирам живут в JSON API):

| метрика | тип | метки | значение |
|---|---|---|---|
| `swg_panel_up` | gauge | — | 1, когда панель отвечает |
| `swg_panel_build_info` | gauge | `version` | информация о сборке (значение 1) |
| `swg_nodes_total` / `swg_nodes_online` | gauge | — | счётчики узлов флота |
| `swg_peers_total` / `swg_peers_online` | gauge | — | счётчики пиров флота |
| `swg_fleet_rx_bytes_per_second` / `…_tx_…` | gauge | — | агрегированная пропускная способность флота |
| `swg_node_up` | gauge | `node`,`name` | 1 online / 0 offline |
| `swg_node_peers` / `swg_node_peers_online` | gauge | `node`,`name` | счётчики пиров по узлу |
| `swg_node_rx_bytes_per_second` / `…_tx_…` | gauge | `node`,`name` | пропускная способность по узлу |
| `swg_node_last_seen_timestamp_seconds` | gauge | `node`,`name` | последняя синхронизация (unix) |
| `swg_node_cpu_percent` / `_memory_percent` / `_disk_percent` | gauge | `node`,`name` | здоровье хоста |
| `swg_node_uptime_seconds` | gauge | `node`,`name` | uptime хоста |

**Webhook'и.** Панель POST'ит подписанное JSON-тело на каждый настроенный URL, когда срабатывает подписанное событие: `peer.added`, `peer.removed`, `node.online`, `node.offline`.

```json
{ "event": "node.offline", "ts": 1783620181,
  "data": { "id": "1b8e0bcb0b4c", "name": "moscow-1", "last_seen": 1783620090 } }
```

Каждая доставка несёт заголовок подписи, чтобы вы могли убедиться, что она пришла от панели:

```
X-SWG-Signature: sha256=<hex HMAC-SHA256(secret, raw_body)>
```

Секрет генерируется при добавлении webhook'а (показывается один раз). Доставка — best-effort с одной повторной попыткой; online/offline узла выводится из устаревания синхронизаций, поэтому `node.offline` срабатывает, когда узел пропускает синхронизации дольше окна offline (`NODE_OFFLINE`, 30с). Наблюдатель **вообще ничего не делает**, когда webhook'и не настроены.

**Примеры:**

```bash
# живучесть (без auth)
curl -s https://panel.example.com/api/v1/health

# флот, с токеном
curl -s -H 'Authorization: Bearer swgp_…' https://panel.example.com/api/v1/servers | jq

# пиры одного узла + тайминг handshake
curl -s -H 'Authorization: Bearer swgp_…' \
  https://panel.example.com/api/v1/servers/moscow-1/peers | jq
```

Конфиг scrape для Prometheus:

```yaml
scrape_configs:
  - job_name: swg-panel
    metrics_path: /metrics
    authorization:
      credentials: swgp_…            # ваш API-токен
    static_configs:
      - targets: ['panel.example.com']
```

Uptime Kuma: добавьте монитор **HTTP(s) - Keyword** на `/api/v1/health`, ключевое слово `"ok"`. Terraform/Ansible: читайте `/api/v1/servers` и `/api/v1/peers`, чтобы обнаруживать состояние флота для инвентаризации или проверок дрейфа (API только для чтения — вносите изменения через UI/ростер панели).

Полный справочник: [`docs/API.md`](docs/API.md).

## Безопасность

- **Транспорт и идентичность панели:** узлы всегда подключаются только *наружу* к панели по TLS. Панель с **настоящим CA** проверяется по системному хранилищу доверия; **самоподписанная** — **автоматически пинится (trust-on-first-use)** при регистрации, так что MITM не сможет выдать себя за неё. Если панель позже сменит адрес (host/port), узел **автоматически перенаправится** на новый только если тот предъявляет запиненный/доверенный сертификат.
- **Подпись запросов (устойчива к replay):** каждый запрос узел→панель несёт `X-SWG-TS` + `X-SWG-MAC` = HMAC-SHA256(`token_sha` узла, `ts`·sha256(body)); панель проверяет его в узком временном окне (отбрасывая повторы) и находит узел по индексу `token_sha` за постоянное время — поэтому поддельный токен не может вызвать усиление pbkdf2.
- **Токены узлов** аутентифицируют узел перед панелью. Они показываются один раз, хранятся только как хеш и могут быть ротированы. Относитесь к `config.json` узла (в нём живой токен) как к секрету — у него режим `600`.
- **PSK** генерируются для каждого пира и хранятся в ростере, чтобы все узлы оставались согласованными; держите `/var/lib/swg-panel` читаемым только пользователем панели.
- **Приватные ключи** генерируются у вас в браузере и никогда не отправляются на сервер (если вы намеренно не включили `store_configs`). Если ключ пира потерян, перевыпустите пира.
- **API-токены** только для чтения и хешируются на диске (SHA-256); утёкший токен может наблюдать состояние флота, но никогда его не изменит, а отключение API в **Settings → Integrations** немедленно отзывает все токены.

## Диагностика

- **Узел остаётся в «awaiting enroll» / никогда не подключается.** Он ещё не синхронизировался. Проверьте `journalctl -u swg-noded -f` на узле: `fingerprint mismatch` означает неверный пин; HTTP `401` — неверный или ротированный токен; ошибка соединения — неверный URL/порт панели или закрытый файрвол.
- **Узел «offline».** Он синхронизировался раньше, но замолчал — проверьте демон и исходящую сеть узла.
- **Пиры не появляются на узле.** Убедитесь, что пир назначен этому узлу и что его интерфейс совпадает с тем, который узел действительно обслуживает. Чтение берётся из последнего снимка узла, поэтому только что добавленному узлу нужна сначала одна синхронизация.
- **Браузер предупреждает о сертификате.** Ожидаемо с `selfsigned`. Используйте настоящий сертификат для продакшена или примите предупреждение за доверенным туннелем.
- **Клиент подключается (handshake проходит), но нет интернета / `rx` растёт, а `tx` остаётся ~0.** Узел расшифровывает пакеты, но не маршрутизирует их наружу. Интерфейсы, созданные установщиком, настраивают это автоматически; для интерфейса, собранного вручную, включите форвардинг (`net.ipv4.ip_forward=1`) и добавьте masquerade для туннельной подсети на WAN-интерфейс (`iptables -t nat -A POSTROUTING -s <subnet> -o <wan> -j MASQUERADE`).
- **Умная маршрутизация переключается не мгновенно — дайте минуту-другую.** Включение списка/категории или смена его выхода — *eventually consistent*, а не мгновенно. Узел применяет маршрутизацию на периодическом reconcile (~60с) после получения разрешённого списка от панели, поэтому есть небольшая задержка, прежде чем совпавший трафик начнёт уходить через новый выход — и дольше, если соединение уже открыто, так как оно продолжает идти по-старому до переподключения. Атрибуция категорий отстаёт чуть сильнее: IP-набор назначения наполняется лениво (по мере разрешения/наблюдения его доменов), поэтому свежемаршрутизированный трафик какое-то время может показываться как **uncategorized**, прежде чем попадёт в свою категорию. Оба сходятся сами; ничего не сломано — перепроверка через минуту или переподключение клиента ускоряет это.
- **`acme.sh` не смог выпустить сертификат.** `letsencrypt` требует доступного порта 80; если 80 занят, используйте `cloudflare` (DNS-01, без порта 80).
